1. El problema que resuelve MermaOps

Contexto, impacto económico y por qué la IA es la solución

📊 El dato que lo explica todo:
España desperdicia 3,1 millones de toneladas de alimentos al año, equivalente a 10 kg por persona. Un supermercado mediano pierde entre 40.000 y 80.000 € anuales solo en merma alimentaria — productos que caducan en estantes, en almacén, o que se deterioran antes de venderse.

¿A quién afecta y por qué es urgente?

Afecta directamente a supermercados independientes, cooperativas y cadenas regionales como Bonarea, Consum o Eroski. A diferencia de Mercadona o Lidl, estas empresas no tienen departamentos de data science propios. La merma no solo es económica: tiene impacto ambiental medible (CO₂, agua), consecuencias legales (CSRD 2025) y un coste reputacional creciente.

€80K Pérdida media anual por tienda
2-5% De ingresos perdidos en merma
90% De la merma es evitable
35% Deducción fiscal por donación (Ley 49/2002)

¿Cómo se resolvía antes?

De forma completamente manual: el encargado hacía rondas físicas, revisaba fechas de caducidad a ojo, y tomaba decisiones de forma reactiva — cuando el producto ya estaba caducado o a punto de caducar. No había predicción, no había trazabilidad, no había automatización.

Soluciones existentes como Winnow o Leanpath requieren hardware propio (básculas inteligentes), costosos contratos de integración, y están orientadas a grandes cadenas. No existe nada accesible para el supermercado de barrio.

"MermaOps lleva la inteligencia de un departamento de data science completo al bolsillo del encargado de turno, a través de Telegram."

2. Arquitectura del sistema

14 agentes especializados coordinados por Kuine

El principio de diseño: especialización + coordinación

En lugar de un único modelo que lo hace todo, MermaOps distribuye la inteligencia entre agentes especializados. Cada agente tiene un modelo óptimo para su tarea: Opus 4.7 para decisiones complejas, Sonnet 4.6 para análisis, Haiku 4.5 para cálculos simples. Esto reduce costes un ~70% respecto a usar el modelo más potente para todo.

Jerarquía de agentes

🧠 Kuine (Orchestrator)
Claude Opus 4.7
Cerebro central. Loop agéntico de hasta 20 iteraciones, 16 tools conectadas. Toma todas las decisiones críticas: rebajar, donar, retirar. Usa extended thinking en casos complejos.
💬 Chuwi (Telegram)
Claude Sonnet 4.6
Agente conversacional del encargado. Streaming progresivo, memoria episódica entre sesiones, proactividad: monitoriza y avisa sin que le pregunten.
📊 Evaluador
Claude Sonnet 4.6
Calcula score de riesgo 0-100 por lote. Activa extended thinking cuando score ≥ 65. Considera caducidad, temperatura, historial del proveedor y factores climáticos.
🛡️ Validador
Claude Sonnet 4.6
Agente adversarial. Bloquea el 100% de 23 tipos de ataque probados: prompt injection, jailbreak, manipulación de datos, bypass de autorización.
🗳️ Consenso
Claude Sonnet 4.6
Lanza 3 instancias paralelas para decisiones críticas. Vota por mayoría. Elimina sesgos de una única instancia en casos de alta incertidumbre.
🔮 Predictor
Claude Haiku 4.5
Predicción de merma a 5 días. Integra meteorología de Open-Meteo (temperatura, lluvia) con historial de ventas para anticipar riesgos.
👁️ Visión
Claude Sonnet 4.6
Análisis de fotos de productos. Detecta fecha de caducidad visible, estado visual y nivel de deterioro. Output JSON estructurado con confianza y acción recomendada.
📰 Reportero
Claude Opus 4.7
Genera briefs diarios, informes semanales y mensuales. Usa Citations API para referenciar exactamente qué dato sustenta cada afirmación.
🌱 ESG
Claude Haiku 4.5
Calcula CO₂ evitado, agua ahorrada, valor recuperado y deducción fiscal por donaciones (Ley 49/2002). Genera puntuación ESG 0-100.
🏷️ Precio
Claude Haiku 4.5
Calcula el descuento óptimo para liquidar stock próximo a caducar, maximizando recuperación de valor sin canibalizar ventas regulares.
📦 Stock
Claude Haiku 4.5
Decisiones de reposición FEFO (First Expired, First Out). Coordina movimientos almacén → tienda según urgencia de caducidad.
🔔 Notificador
—
Orquesta las alertas proactivas de Telegram: botones inline para donar, rebajar o retirar directamente desde el chat, sin abrir la app.

¿Por qué esa división de agentes?

Tres principios guiaron el diseño:
  • Coste de razonamiento: Opus para decisiones que afectan a dinero real, Haiku para aritmética simple. Un euro de API bien gastado rinde más que diez en el modelo equivocado.
  • Paralelismo real: El Evaluador, Validador y Consenso pueden correr en paralelo sobre el mismo lote. Kuine los orquesta y combina los resultados.
  • Separación de responsabilidades: Cada agente tiene un único objetivo medible. Si el Validador falla, el problema está aislado. Si el Predictor es impreciso, no afecta a las decisiones operativas del día.

4. ¿Qué he construido?

Componentes técnicos, stack y decisiones de implementación

Backend FastAPI + Python 3.14
Puerto 8001. 60+ endpoints REST. APScheduler para jobs. Langfuse para observabilidad de tokens y costes. Rate limiter para proteger API.
Claude API (Anthropic)
Opus 4.7 para Kuine y Reportero. Sonnet 4.6 para Chuwi, Evaluador, Validador. Haiku 4.5 para Precio, Stock, ESG, Predictor. Tool use, streaming, extended thinking, Citations API.
Supabase (PostgreSQL + Auth + Realtime)
28 tablas: batches, actions, merma_log, agent_conversations, agent_messages, supervisor_decisions, telegram_users... Auth JWT para Flutter y backend.
Flutter App (iOS/Android/Web)
6 pantallas principales: Dashboard, Scan (barcode), Acciones, Mapa, Informes (9 tabs), Agentes (4 tabs). Riverpod, GoRouter, notificaciones locales.
Telegram Bot (@ChuwiMermaOpsBot)
python-telegram-bot. Streaming progresivo de respuestas. Botones inline para accionar directamente. Acepta texto, fotos y notas de voz.
Testing: 800 tests en <2s
Pytest con mocks de Supabase y Claude. Sin conexión real en CI. Cobertura de rutas, agentes, evaluador, validador (23 ataques adversariales) y scheduler.

8. Reflexión final y aprendizajes

Qué funcionó, qué mejoraría y qué me llevo

¿Qué ha funcionado bien?

  • La arquitectura multi-agente funcionó exactamente como estaba diseñada: especializar agentes por tarea reduce costes y aumenta la calidad. Los 800 tests lo validan.
  • Telegram como interfaz principal fue el acierto más grande del proyecto. Eliminar la fricción de instalar una app nueva fue clave para la adoptabilidad real.
  • El sistema adversarial demostró ser robusto: 100% de ataques bloqueados sin necesidad de reglas manuales — el Validador razona sobre la intención.
  • La proactividad de Chuwi cambia el modelo mental: pasar de "herramienta que responde" a "agente que monitoriza" es el salto conceptual más importante del proyecto.

¿Qué mejoraría?

  • Agente de compras con wallet: conectar Kuine a un sistema de pago para ejecutar pedidos automáticos a proveedores cuando el stock baja. La arquitectura ya está lista — solo falta la integración de pagos.
  • Inteligencia multi-tienda: una cadena con 10 tiendas tipo Bonarea podría comparar qué vende más en cada ubicación (más pollo en una calle, más carne en otra) y optimizar el pedido por tienda.
  • Integración con TPV real: conectar el sistema de caja para saber ventas en tiempo real, no solo merma. Cierra el loop entre lo que entra y lo que sale.
  • RLS de Supabase para producción: actualmente usa service key para demo. En producción, habría que implementar Row Level Security por tienda.

Aprendizajes del proceso

La diferencia entre un bot y un agente es la autonomía

Un bot responde preguntas. Un agente actúa. Chuwi no espera que le pregunten — detecta cambios, decide si son relevantes y avisa. Kuine no sigue un script — razona con sus herramientas y llega a conclusiones propias. Construir esa diferencia en código, con modelos reales, es la lección central de este módulo.

Tres cosas que aprendí y no esperaba:
  1. El diseño del prompt es arquitectura. Cómo describes el rol de Kuine determina la calidad de sus decisiones más que el modelo que elijas. Un prompt mal diseñado con Opus vale menos que uno bien diseñado con Haiku.
  2. Los agentes fallan de formas interesantes. El Evaluador sobreestimaba el riesgo los viernes porque el historial de ventas del fin de semana era menor. Aprender a calibrar esos sesgos es parte del trabajo del diseñador de sistemas agénticos.
  3. La confianza del usuario es el sistema más difícil de diseñar. Cuando Kuine dice "rebaja esta merluza un 40%", el encargado tiene que confiar en esa recomendación. Construir esa confianza requiere trazabilidad, explicabilidad y un historial de aciertos — no solo precisión técnica.
